## Redis
Redis是一个开源（BSD许可）的内存型数据结构存储；常用作数据库，缓存和消息代理。  

支持的数据结构：
- String
- Hash
- List
- Set
- Sorted Set(带有范围查询的排序集)

除了以上五种数据结构，Redis还支持 
- bitmap
- hyperloglog
- geospatial（半径查询和流的地理空间索引）

这几个都没用到过，以后有用到在补充

Redis具有内置复制，Lua脚本，LRU驱逐，事务和不同级别的磁盘持久性，并通过Redis Sentinel提供高可用性并使用Redis Cluster自动分区。

以上是Redis的基本介绍；具体可以参考[官网](https://redis.io/topics/introduction)
### Redis 有什么优缺点？
#### 优点
- 速度快： 因为Redis的数据存在内存中，内存的读取速度更快；也支持数据持久化到磁盘中。
    + 持久化： 可以通过每隔一段时间将数据集转储到磁盘或记录每一条命令到日志来持久化。
- 支持丰富的数据类型： String，List,Hash,Set, SotedSet
- 支持事务： 所有的操作都是原子性的，也支持对几个操作合并后的原子执行（MULTI, EXEC, DISCARD and WATCH）。
- 丰富的特性 

### Redis 缓存失效策略
- noeviction(不驱逐)： 当达到内存限制，但是客户端尝试执行可能需要更多内存的指令时返回错误
- allkeys-lru： 首先移除最近最少使用的键，为新增数据腾出空间
- volatile-lru： 首先从已经设置过期时间的键中移除最近最少使用的键，为新增数据腾出空间。 
- allkeys-random： 从所有键中随机移除，为新增数据腾出空间
- volatile-random： 从已经设置过期时间的键中随机移除某个key
- volatile-ttl： 在设置了过期时间的key中，首先移除有更早过期时间的key

### Redis 主键失效策略
- 惰性： 当key过期后不管，当客户端访问这个key时，使key过期，并且发现key超时
- 定期过期： 因为过期的key可能永远不会被访问到，所以需要定期过期处理；redis会定期的在设置了过期时间的key中随机测试key；
	1. 从过期密集的keys中随机测试20个keys
	2. 删除发现的已过期的key
	3. 如果过期数超过25%，重复1  
这意味着在任何给定时刻，使用内存的已经过期的键数量最大等于每秒最大写入操作量除以4。

为了在不牺牲一致性的情况下获得正确的行为，当密钥到期时，在AOF文件中合成DEL操作并获得所有附加的副本节点。

- 立即清理（官网？）：在设置键的过期时间时，创建一个回调事件，当过期时间达到时，由时间处理器自动执行键的删除操作。

### 缓存雪崩，缓存穿透
#### 缓存穿透
缓存指的是查询根本不存在的数据，在缓存和存储层都不会命中。  
因为一般情况下，如果存储层没有查到数据，则不会写入缓存中，这就导致不存在的数据每次都需要到存储层查询。如果是恶意攻击，会导致存储层的负载增大，也失去了缓存的意义。

##### 缓存穿透的解决方法
- 互斥锁排队： 如果缓存中不存在，则排队，到存储层查询
- 缓存空值，并设置一个短时间的有效期；如果数据没有查询到，可以将null写入到缓存中。但是这样可能会影响正常的业务，所以需要设置一个较短的过期时间。
- 布隆过滤器拦截： 在访问缓存层和存储层之前，将存在的 key 用布隆过滤器提前保存起来，做第一层拦截。如果key在布隆过滤器不命中，则直接返回，如果存在，就查询缓存或者存储层

#### 缓存雪崩
大量缓存在同一时间过期或者缓存不可用，导致大量数据查询直接到存储层，造成存储层负载过高。

##### 缓存雪崩的解决方法
- 缓存数据的过期时间设置随机，防止同一时间大量数据过期现象发生。
- 设置热点数据永不过期或者过期时间长一些
- 建立备份缓存
